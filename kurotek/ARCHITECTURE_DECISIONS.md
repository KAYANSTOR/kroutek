# سجل القرارات المعمارية (Architecture Decision Records - ADR)

هذا الملف يوثق جميع القرارات الاستراتيجية والمعمارية لتطبيق **Kurotek**. يُمنع منعاً باتاً اتخاذ أو تنفيذ أي قرارات متعارضة مع ما هو مدون هنا من قبل أي مطور أو ذكاء اصطناعي.

## ADR-001: Server First Architecture
- **الحالة:** معتمد
- **الوصف:** جميع مصادر الحقيقة (Source of Truth) وصلاحيات التراخيص وتأكيدات العمليات تأتي من الخادم. التطبيق يتبع الخادم وليس العكس.

## ADR-002: Offline First Support
- **الحالة:** معتمد
- **الوصف:** على الرغم من أن الخادم هو الأساس (ADR-001)، التطبيق مصمم للعمل بسلاسة في حال انقطاع الإنترنت (Offline First). يتم تسجيل الحركات والتراخيص محلياً في قواعد بيانات مشفرة ومزامنتها فور عودة الاتصال باستخدام Sync Engine.

## ADR-003: No Firebase Database for Core Logic
- **الحالة:** معتمد
- **الوصف:** يُمنع استخدام Firebase (Firestore/Realtime Database) كقاعدة بيانات أساسية لمنطق الأعمال والتراخيص. يمكن استخدام خدمات Firebase فقط للإشعارات (FCM) أو التحليلات (Analytics) بشكل جانبي إن لزم الأمر.

## ADR-004: Node.js / TypeScript Backend (~~Go~~ — مُعدَّل)
- **الحالة:** معتمد (مُعدَّل بقرار صريح من صاحب المشروع بتاريخ هذه الجلسة)
- **القرار السابق (مُلغى)**: كان هذا البند ينص على بناء الخادم بلغة Go. لم يُنفَّذ أي كود Go فعلياً، بينما كان يتعارض مباشرة مع خادم TypeScript/Node.js/Express/Prisma الكامل والمنفَّذ فعلياً (`server/src/`، انظر `docs/RESTRUCTURING_PLAN.md` القسم 5.5 والقسم 9 البندين 8 و10) — بما يشمل: بنية Modules كاملة، `schema.prisma` مُختبَر على PostgreSQL حقيقي، ومحرك مزامنة REST (Push/Pull) بمنطق Last-Write-Wins.
- **الوصف (الحالي)**: البنية التحتية للخادم مبنية بـ **Node.js + TypeScript + Express + Prisma**، لا Go. جميع طبقات الـ API في التطبيق (`ApiEndpoints.kt` وما يتصل بها) يجب أن تتوافق مع مسارات الخادم TypeScript الفعلية الموجودة في `server/src/`، لا العكس. **أي تعليق كود أو توثيق يذكر "Go Backend" يُعتبر مخالفاً لهذا القرار المُحدَّث ويجب تصحيحه.**
- **السبب**: تفادي ازدواجية عمل حقيقية بين خادمين بلغتين مختلفتين لنفس الغرض؛ الخادم TypeScript منفَّذ ومُختبَر جزئياً فعلياً بينما Go لم يكن له أي تنفيذ.

## ADR-005: PostgreSQL Database
- **الحالة:** معتمد
- **الوصف:** قاعدة البيانات الأساسية في الخادم هي PostgreSQL. جميع هياكل البيانات (Data Models/DTOs) في التطبيق يجب أن تكون متسقة وتدعم أنواع بيانات علائقية متوافقة مع PostgreSQL (مثل UUIDs, Timestamps).

## ADR-006: Repository & Clean Architecture Pattern
- **الحالة:** معتمد
- **الوصف:** التطبيق يتبع هيكلية معمارية نظيفة وصارمة. التسلسل المسموح للأوامر هو حصراً:
  `UI ➔ ViewModel ➔ UseCase ➔ Repository ➔ Engine ➔ API / Local Database`
  ويُمنع تماماً تواجد أي (Business Logic) داخل ViewModels أو واجهات الـ UI.

## ADR-007: Strict Separation of DTOs and Domain Models
- **الحالة:** معتمد
- **الوصف:** تُمنع واجهات المستخدم (UI) من التعامل المباشر مع كائنات نقل البيانات (DTOs). طبقة Mapper هي المسؤولة عن تحويل الـ DTOs القادمة من الـ API إلى Domain Models والعكس.

## ADR-008: Independent Sync Engine
- **الحالة:** معتمد
- **الوصف:** يتم بناء محرك مزامنة (Sync Engine) مستقل تماماً ومبني بخصائص متقدمة (Queue, Delta, Retry, Conflict Resolution). يُستخدم `WorkManager` فقط كـ Scheduler (مجدول) لتشغيل محرك المزامنة وليس لكتابة منطق المزامنة بداخله.
